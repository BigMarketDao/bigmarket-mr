# docs-v2 gap report

> Read-only audit produced from the prompt in `[docs-v2/audit-prompt.md](./audit-prompt.md)`.
> Verdicts are **Covered / Thin / Missing / Deferrable / Unknown**. Citations are `path:lines` or "not present".
> No files in `docs-v2/` or the codebase were modified.

---

## 1. Summary verdict

**Today `docs-v2/` is good enough to land the *plumbing* of a tokenized UI, but it is not good enough — on its own — to ship a Polymarket-grade prediction-market product.** The pack is strong on contracts, naming, and migration order, and it correctly identifies the Skeleton-secondary ↔ accent trap. But it is silent on every prediction-market-specific surface that the user will actually look at: market cards, trade widget, order book, chart tokens, numeric formatting, live-tick behavior, pending-tx state, address pills, toast/notifications, and skeleton/empty/error states. The "visual baseline" today is Skeleton `nouveau` + the empty Polymarket method template — there is no measured reference UI to push against. Meanwhile the runtime is already consuming **eight "Next wave" tokens** (`bg-card`, `bg-popover`, `bg-input`, `border-input`, etc.) in `bm-ui` components that have **no backing variable in `theme.css`**, so several UI primitives are currently rendering with broken or inherited values — and the docs do not flag this.

**Top three blockers for shipping a visibly more professional product on docs-v2 alone:**

1. **No prediction-market component spec.** Market card variants, trade widget anatomy, order book, KPI strip, position card, pending-tx chip — none are described or even named anywhere in `docs-v2/`. An agent will invent them.
2. **No measured baseline.** `[docs-v2/design/polymarket-ui-audit.md](./design/polymarket-ui-audit.md)` is a method with empty result tables (lines 127–139) and the screenshots in `docs/design/assets/polymarket/` are referenced from a doc but never used as the source of truth. There is no numeric "this is the radius we ship" / "this is the muted text color" anchor.
3. **Token wave 1 is consumed before it is wired.** `bg-card`, `bg-popover`, `text-popover-foreground`, `bg-input`, `border-input` are already used in `packages/bm-ui/` but are listed **Next wave** in `[docs-v2/design/bm-semantic-tokens-proposal.md](./design/bm-semantic-tokens-proposal.md)` and are **absent from `[packages/bm-design/src/theme.css](../packages/bm-design/src/theme.css)`** (no `--color-card` / `--color-popover` / `--color-input` semantic alias). The docs do not warn about this drift.

A careful agent could ship a *competent* refactor of the chrome with what is here, but not "Polymarket / Kalshi". To get there, docs-v2 needs (in priority order): a populated Polymarket baseline, a market-component spec, a numeric-formatting spec, a domain/status token wave that actually compiles, a chart-token + library decision, and a one-page interaction/a11y spec.
